# Правила ведения журнала изменений

В проекте `Calk_KMF` каждое значимое изменение фиксируется в Git с детальным комментарием. Это позволяет:

- понимать историю изменений на GitHub;
- быстро находить, когда и почему было внесено изменение;
- формировать `CHANGELOG.md` и релиз-ноты;
- упрощать код-ревью и восстановление контекста.

## Шаблон коммита

При выполнении `git commit` без флага `-m` автоматически открывается шаблон из файла `.gitmessage`.

Структура сообщения:

```
[ТИП] Краткое описание изменения (до 72 символов)

Подробное описание:
- Что изменено и почему
- Какие модули затронуты
- Результаты тестирования (если есть)
- Ссылки на связанные issue/документы
```

## Типы изменений

| Тип      | Описание                                          |
|----------|---------------------------------------------------|
| `[ADD]`  | Новая функциональность                            |
| `[FIX]`  | Исправление ошибки                                |
| `[REFACT]` | Рефакторинг без изменения поведения             |
| `[PERF]` | Улучшение производительности                      |
| `[TEST]` | Добавление или обновление тестов                  |
| `[DOCS]` | Изменения в документации                          |
| `[DB]`   | Изменения схемы или миграции базы данных          |
| `[CONFIG]` | Настройки, зависимости, CI/CD                   |

## Примеры

### Плохо

```
fix
```

### Хорошо

```
[FIX] Исправить расчёт day_pillar для даты смены месяца

В `code/bazi_calendar/pillars.py` скорректирована логика определения
дня смены столпа в момент между 23:00 и полуночью по местному времени.
Добавлены контрольные тесты для граничных дат 2026-02-03 и 2026-02-04.

Тесты:
- code/tests/test_pillars.py: 8/8 passed
- Сверка с control_pillars_utc.csv
```

## Порядок работы

1. Перед коммитом убедись, что изменения протестированы.
2. Выполни `git add <files>`.
3. Выполни `git commit` (без `-m`), чтобы открылся редактор с шаблоном.
4. Заполни все разделы шаблона.
5. Отправь изменения на GitHub: `git push origin main`.

## CHANGELOG.md

Крупные изменения дублируются в `CHANGELOG.md` в секции `[Unreleased]`.
Перед релизом секция `[Unreleased]` преобразуется в номер версии с датой.
