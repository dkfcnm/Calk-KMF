# Протокол взаимодействия «Пользователь — AI-агент»

**Дата вступления в силу:** 2026-05-24  
**Версия:** 1.1  
**Статус:** Активен

---

## 1. Цель протокола

Обеспечить предсказуемость, однозначность и отслеживаемость коммуникации между пользователем (заказчиком) и AI-агентом (исполнителем) при постановке, уточнении и реализации задач в проекте Calk_KMF.

---

## 2. Жизненный цикл задачи

```
Постановка → Анализ → Уточнение → Планирование → Реализация → Тестирование → Приёмка
```

### 2.1. Постановка
Пользователь описывает задачу естественным языком. Форма произвольная, но желательно включать:
- Что нужно сделать
- Зачем (контекст / бизнес-цель)
- Ограничения или особые требования
- Приоритет

### 2.2. Анализ (обязательный этап)
AI-агент **перед началом любой реализации** проводит анализ задачи и проверяет:

1. **Полнота:** все ли детали задачи понятны?
2. **Однозначность:** нет ли двусмысленных формулировок?
3. **Варианты:** существует ли несколько способов решения?
4. **Риски:** есть ли скрытые риски, которые могут привести к проблемам?

Если хотя бы по одному пункту есть сомнения — задача **не переходит к реализации** до получения исчерпывающих ответов.

> **Исключение (Р.1 — принято):** Для задач, которые явно укладываются в ≤3 минут работы, не затрагивают архитектуру и не влияют на другие модули, анализ считается мгновенным. AI-агент может переходить к реализации без формулировки дополнительных `В.N`.

### 2.3. Уточнение
AI-агент формулирует запросы пользователю по строгой схеме маркировки (см. раздел 3).

### 2.4. Планирование
После устранения всех неясностей AI-агент формирует план реализации. Если план содержит альтернативные подходы — они выносятся как **Предложения** (П.N) для одобрения пользователя.

#### 2.4.1. Параллельное выполнение задач
AI-агент оценивает **весь скоп задач**, полученных от пользователя в рамках одной сессии или контекста. Если несколько задач:
- независимы друг от друга,
- не конфликтуют за одни и те же файлы,
- могут быть выполнены параллельно без ущерба качеству,

— AI-агент **обязан предложить** пользователю параллельное выполнение как `П.N` с обоснованием (экономия времени, отсутствие блокировок). Параллельное выполнение начинается только после явного согласия пользователя.

### 2.5. Реализация
Выполняется только после утверждения плана и однозначного выбора вариантов.

### 2.6. Тестирование
AI-агент обязан проверить результат (компиляция, тесты, визуальная проверка) и сообщить о статусе.

### 2.7. Приёмка
Пользователь принимает или возвращает на доработку с комментариями.

## 3. Схема маркировки запросов

Каждый запрос от AI-агента к пользователю в рамках этапа уточнения маркируется заглавной буквой, точкой и порядковым номером. Это позволяет точечно отвечать на конкретные пункты.

| Тип запроса | Маркировка | Пример | Когда применять |
|-------------|------------|--------|-----------------|
| **Вопрос** | `В.N` | В.1, В.2 | Неясность, недостающая информация, необходимость уточнения формулировки |
| **Предложение** | `П.N` | П.1, П.2 | Рекомендация по реализации, альтернативный подход, улучшение |
| **Риск** | `Р.N` | Р.1, Р.2 | Потенциальная проблема, технический долг, конфликт с существующим кодом |

### 3.1. Правила формулировки

**Вопрос (В.N):**
- Чётко формулируй, чего не хватает для продолжения
- Если возможно, предложи варианты ответа

**Предложение (П.N):**
- Опиши предлагаемое решение
- Перечисли плюсы и минусы
- Если есть альтернативы — укажи их как отдельные предложения

**Риск (Р.N):**
- Опиши потенциальную проблему
- Укажи вероятность и влияние (если оцениваемы)
- Предложи митигацию (способ снижения риска)

---

## 4. Обработка ответов пользователя

### 4.1. Прямые ответы
Если пользователь отвечает на `В.N`, `П.N` или `Р.N` — AI-агент:
1. Фиксирует ответ в контексте
2. Если ответ содержит **поручение** (новая задача, изменение требований, откладывание) — записывает его в `PROJECT.md` в раздел «Поручения»
3. Если после ответа остаются незакрытые пункты — продолжает уточнение
4. Если все пункты закрыты — переходит к планированию / реализации

### 4.2. Отложенные пункты
Если пользователь говорит: *«Отложи В.3»*, *«П.2 пока не делаем»*, *«Р.1 оставим на потом»* — AI-агент:
1. Не включает отложенный пункт в текущую реализацию
2. **Обязательно записывает** его в `PROJECT.md` в соответствующий раздел:
   - Отложенные вопросы → `## Отложенные вопросы`
   - Отложенные предложения → `## Отложенные предложения`
   - Отложенные риски → `## Отложенные риски`
3. При постановке следующей связанной задачи — напоминает об отложенных пунктах.
4. **(Р.2 — принято)** При старте каждой новой сессии AI-агент проверяет отложенные разделы `PROJECT.md` и напоминает о релевантных пунктах, если они связаны с текущим контекстом.

### 4.3. Поручения
Любая фраза вида *«Сделай X»*, *«Запиши Y»*, *«Добавь Z»* в ответе пользователя трактуется как поручение и фиксируется в `PROJECT.md` в разделе `## Поручения`.

---

## 5. Запрещённые действия (stop-правила)

AI-агент **НЕ имеет права**:

1. Начинать реализацию, пока есть открытые `В.N` без ответа
2. Выбирать вариант реализации самостоятельно, если пользователь не дал однозначного ответа на соответствующее `П.N`
3. Игнорировать озвученный `Р.N` без согласования с пользователем
4. Забывать отложенные пункты — они должны быть в `PROJECT.md`
5. Вносить изменения в production-базы данных без явного разрешения

---

## 6. Обновление протокола

Изменения в протокол вносятся по инициативе пользователя или по согласованию сторон. Версия протокола указывается в заголовке файла.
