# Оптимизация Анализа Прогулок Ци Мэнь (Qimen Walk Analysis)

## 1. Проблема
Предыдущая реализация анализа прогулок (`run_analysis.py`) работала крайне медленно (часы) и была склонна к зависаниям из-за:
- Построчной обработки данных в Python (1M+ строк).
- Неэффективных запросов к БД (N+1 проблема).
- Линейного поиска правил для каждой карты.

## 2. Реализованное Решение (Optimized Engine)
Мы внедрили гибридный подход с использованием возможностей SQL и оптимизированного движка правил.

### Архитектура
1. **RuleEngine (Python):**
   - **Индексация правил:** Правила теперь загружаются в хэш-карты (Lookup Maps) по ключам `(HeavenStem, EarthStem)`, `Gate`, `Star`.
   - **Поиск O(1):** Вместо перебора всех правил, для каждой карты проверяются только те, которые соответствуют её элементам.

2. **Data Fetching (SQL):**
   - Используются **раздельные SQL-запросы** для наполнения каждой из 3 таблиц результатов (`Date`, `Direction`, `Activation`).
   - `PRAGMA synchronous = OFF` и `journal_mode = MEMORY` ускоряют запись результатов.
   - Пакетная вставка (Batch Insert) или прямая вставка через `INSERT INTO ... SELECT`.

### Результаты
- Время выполнения полного анализа сократилось до ~2-5 секунд.
- Устранено потребление памяти, связанное с накоплением объектов.
- Результаты корректно разделены по типу (Дата vs Направление).

## 3. Структура Таблиц Результатов

Результаты анализа разделены на 3 независимые таблицы в соответствии с областью применения правил.

### `t_analysis_date` (Анализ Даты)
Хранит результаты оценки качества времени (Бацзы, Тун Шу, Офицеры и т.д.), не зависящие от направления.
- **PK:** `(hour_id, rule_id)`

| Column | Type | Description |
|--------|------|-------------|
| hour_id | TEXT | Ссылка на час (из `t_bazi_hourly`) |
| rule_id | TEXT | Сработавшее правило (из `spr_rule_registry`) |
| result_value | TEXT | Текстовое описание результата (например, название звезды) |
| score | REAL | Уровень благоприятности (число) |

### `t_analysis_direction` (Анализ Направления / Расклада)
Хранит результаты оценки раскладов Ци Мэнь (структуры, врата, звезды в конкретных дворцах).
- **PK:** `(hour_id, palace_no, system_type, chart_level, rule_id)`

| Column | Type | Description |
|--------|------|-------------|
| hour_id | TEXT | Ссылка на время |
| palace_no | INTEGER | Номер дворца (1-9) |
| system_type | TEXT | Система расчета (`ZhiRun`, `ChaiBu`) |
| chart_level | TEXT | Уровень расклада (`Year`, `Month`, `Day`, `Hour`) |
| rule_id | TEXT | Сработавшее правило |
| result_value | TEXT | Текстовое описание (Название структуры) |
| score | REAL | Уровень благоприятности |

### `t_analysis_activation` (Анализ Активаций)
Предназначена для результатов поиска специальных активаций.
- **PK:** `(hour_id, palace_no, rule_id)`

| Column | Type | Description |
|--------|------|-------------|
| hour_id | TEXT | Ссылка на время |
| palace_no | INTEGER | Номер дворца |
| rule_id | TEXT | Сработавшее правило |
| score | REAL | Уровень благоприятности |

## 4. Восстановление и Инструкция по запуску (Restoration Guide)

### Предварительные требования
1. **Разблокировка БД:** Если при запуске `maintenance.py` возникает ошибка `sqlite3.OperationalError: database is locked`, это означает, что предыдущий процесс Python не завершился корректно.
   - **Решение:** Перезагрузите IDE/Терминал или выполните `taskkill /F /IM python.exe`.
2. **Контрольные данные:** После изменения логики (например, исправление имен колонок) необходимо обновить контрольные таблицы.
   - **Команда:** `py code/tests/control_data.py create_all`

### Запуск полного цикла
Для обновления всех данных и запуска анализа используйте:
```powershell
py maintenance.py
```

### Конфигурация запуска
Файл `code/analysis/run_analysis.py` настроен на **SQL-Native Only** режим для структур. 
- Цикл обработки правил Python закомментирован для максимальной скорости.
- Если потребуется добавить сложные Python-правила (например, Офицеры), их нужно раскомментировать или (лучше) реализовать отдельным SQL-запросом.
