# Спецификация расчёта бацзы‑календаря (v0.2)

## 1. Цель и границы

- **Цель**: автоматизировать расчёт бацзы‑календаря (4 столпа: год, месяц, день, час) для каждой часовой точки времени, с прицелом на последующее расширение под Тун Шу и Ци Мэнь.
- **Диапазон расчёта на текущем этапе разработки**:
  - по умолчанию: *текущий год + 2 следующих года*;
  - расчёт за прошлые периоды не требуется.
- **Хранилище**: проект использует файл *SQLite* `calk_kmf.sqlite`. DuckDB рассматривается как возможный целевой вариант, но все актуальные расчёты и справочники находятся в SQLite.
- **Результат**: таблица `t_bazi_hourly` и набор view в SQLite. Представления формируются программно и содержат предвычисленные значения, включая лунный месяц, день и признак високосного месяца, что позволяет быстро обращаться к данным без «на лету» пересчёта.
- **Метаданные**: таблицы `spr_table_comment` и `spr_column_comment` обязательны к заполнению и актуализации при любом изменении схемы; пустых описаний быть не должно.
- **Контроль качества**: перед тем как отчитываться о завершении задачи, исполнитель обязан самостоятельно проверить актуальность расчётных таблиц и представлений (включая производные по часовым поясам) и убедиться, что новые поля/данные действительно доступны.

Пакет `code.bazi_calendar` разделён на четыре подсистемы:

1. `db` — инфраструктура работы с SQLite (подключение, резервное копирование, утилиты UTC).
2. `solar_terms` — доступ к временам солнечных терминов. В приоритете используются данные Гонконгской обсерватории (таблица `t_solar_term_time_hko`); при отсутствии — расчётные значения `t_solar_term_time`.
3. `lunar` — конвертация григорианских дат в китайский лунный календарь, предоставление месяца, дня и признака високосного месяца.
4. `pillars` — чистые функции расчёта столпов года/месяца/дня/часа и агрегирующая `calc_four_pillars`.
5. `hourly` — слой хранения: массовое заполнение `t_bazi_hourly`, создание view `v_bazi_hourly`, `v_bazi_hourly_tz_*`, `v_bazi_hourly_msk`.

Контрольные данные (`data/control_pillars_utc.csv` и таблица `t_control_pillars`) синхронизируются со скриптом `rebuild_control_data.py`, который пересчитывает столпы через `calc_four_pillars` и гарантирует согласованность с приоритетом HKO.

Все методологические правила базируются на документах `bazci_01.md`, `bazci_01_answers_01.md` и контрольных значениях `bazci_control.md`.

## 2. Используемые справочники и структуры данных

### 2.1. Небесные стволы и земные ветви

#### Таблица `spr_heavenly_stem`

Справочник небесных стволов.

- `stem_id` INTEGER PK (1..10)
- `stem_char` TEXT — иероглиф: 甲, 乙, 丙, 丁, 戊, 己, 庚, 辛, 壬, 癸
- `stem_pinyin` TEXT — при необходимости
- `stem_rus` TEXT — русская транскрипция/название
- `element` TEXT — элемент (Дерево, Огонь, Земля, Металл, Вода)
- `yin_yang` TEXT — «Инь» или «Ян»

#### Таблица `spr_earthly_branch`

Справочник земных ветвей (по таблице из `bazci_01.md`).

- `branch_id` INTEGER PK (1..12)
- `branch_char` TEXT — 子, 丑, 寅, 卯, 辰, 巳, 午, 未, 申, 酉, 戌, 亥
- `branch_pinyin` TEXT
- `branch_rus` TEXT — Крыса, Бык, Тигр и т.д.
- `element` TEXT — элемент
- `yin_yang` TEXT — «Инь» или «Ян»

### 2.2. Солнечные сезоны (24 цзеци) и солнечные месяцы

#### Таблица `spr_solar_term`

Справочник 24 солнечных сезонов с долготами и земными ветвями месяца.

- `solar_term_id` INTEGER PK (1..24) — порядковый номер сезона
- `solar_term_char` TEXT — китайское название (立春, 雨水, …)
- `solar_term_name_ru` TEXT — русское название
- `solar_term_pinyin` TEXT — транскрипция
- `longitude_deg` INTEGER — долгота Солнца: 0, 15, 30, …, 345
- `month_branch_id` INTEGER FK → `spr_earthly_branch.branch_id` — земная ветвь месяца, к которой относится сезон

**Правило:**
- 24 сезона делят эклиптику на 24 отрезка по 15°.
- Два последовательных сезона образуют один солнечный месяц.
- Номер солнечного месяца: `month_index = ceil(solar_term_id / 2)` (1..12).

#### Таблица `t_solar_term_time`

Результат астрономических расчётов моментов прохождения Солнцем границ сезонов.

- `year` INTEGER — григорианский год
- `solar_term_id` INTEGER FK → `spr_solar_term`
- `longitude_deg` INTEGER — дублирование для удобства
- `crossing_utc` TIMESTAMP — момент, когда эклиптическая долгота Солнца = `longitude_deg` (UTC)
- `crossing_gmt0` TIMESTAMP — тот же момент, приведённый к GMT+0 (на текущем этапе совпадает с UTC)

PK: (`year`, `solar_term_id`).

**Назначение:**
- определить границы солнечных месяцев;
- определить точный момент наступления китайского нового года (Ли Чунь, 315°);
- реализовать правило «две записи в одном часе при смене сезона»;
- служить резервным источником, если для года нет эталонных данных обсерватории.

#### Таблица `t_solar_term_time_hko`

Справочник эталонных моментов сезонов из Гонконгской обсерватории (HKO) для лет, по которым доступны точные наблюдения.

- `year` INTEGER — календарный год наблюдения;
- `solar_term_id` INTEGER FK → `spr_solar_term`;
- `longitude_deg` INTEGER — проверка соответствия долготе сезона;
- `crossing_hkt` TIMESTAMP — момент сезона в HKT (UTC+8);
- `crossing_utc` TIMESTAMP — тот же момент, приведённый к UTC;
- `tz_offset_hours` INTEGER — смещение часового пояса источника (по умолчанию 8 часов).

PK: (`year`, `solar_term_id`).

**Назначение:**
- переопределять расчётные значения `t_solar_term_time` для лет 2025–2027 (и других, если будут добавлены в будущем);
- обеспечивать точное совпадение с контрольными данными HKO.

### 2.3. Часы (Chen) и земные ветви часов

#### Таблица `spr_hour_branch`

Справочник соответствия 12 двойных часов (Chen) земным ветвям.

Базируется на таблице из `bazci_01.md` (часы в GMT+8):

- `hour_slot_id` INTEGER PK (1..12) — порядковый номер Chen
- `branch_id` INTEGER FK → `spr_earthly_branch`
- `start_hour_gmt8` INTEGER — час начала в GMT+8 (0..23)
- `end_hour_gmt8` INTEGER — час окончания в GMT+8 (0..23), с учётом диапазонов вида 23–1

### 2.4. Правила формирования столпов месяца и часа

#### Таблица `spr_pillar_month_rule`

Стандартная схема: небесный ствол года + номер солнечного месяца → небесный ствол месяца.

- `year_stem_id` INTEGER FK → `spr_heavenly_stem`
- `month_index` INTEGER (1..12) — номер солнечного месяца (1-й месяц Инь и т.п.)
- `month_stem_id` INTEGER FK → `spr_heavenly_stem`

PK: (`year_stem_id`, `month_index`).

#### Таблица `spr_pillar_hour_rule`

Стандартная схема: небесный ствол дня + земная ветвь часа → небесный ствол часа.

- `day_stem_id` INTEGER FK → `spr_heavenly_stem`
- `hour_branch_id` INTEGER FK → `spr_earthly_branch`
- `hour_stem_id` INTEGER FK → `spr_heavenly_stem`

PK: (`day_stem_id`, `hour_branch_id`).

### 2.5. 60‑летний цикл столпов

Для удобства расчётов заводится справочник полного 60‑летнего цикла.

#### Таблица `spr_pillar_cycle`

- `cycle_index` INTEGER PK (0..59) — позиция в цикле Дзя‑Цзы
- `stem_id` INTEGER FK → `spr_heavenly_stem`
- `branch_id` INTEGER FK → `spr_earthly_branch`

Используется для расчёта столпов года и дня по смещению от опорной точки.

### 2.6. Области применения правил

#### Таблица `spr_analysis_scope`

Справочник контекстов, в которых могут использоваться показатели/правила анализа.

- `scope_code` TEXT PK — код области (например, `date`, `activation`, `walk`, `qimen_chart`);
- `name_ru` TEXT — русское название области;
- `description_ru` TEXT — описание применения.

#### Таблицы `spr_indicator_scope`, `spr_indicator_rule_scope`

Связывают показатели (`spr_indicator`) и правила (`spr_indicator_rule`) с областями применения. Каждая строка означает, что показатель/правило может быть задействован в указанном контексте. Первичный ключ составной (`indicator_id`, `scope_code`) и (`rule_id`, `scope_code`).

### 2.7. Справочник 12 офицеров дня

#### Таблица `spr_day_officer_mapping`

Матрица соответствия земных ветвей месяца и дня офицерам из традиционного цикла «12 офицеров».

- `month_branch_id` INTEGER FK → `spr_earthly_branch` — земная ветвь солнечного месяца;
- `day_branch_id` INTEGER FK → `spr_earthly_branch` — земная ветвь дня;
- `officer_value_id` INTEGER FK → `spr_indicator_value` — значение перечислимого показателя `day_officer`.

Каждая пара ветвей (`month_branch_id`, `day_branch_id`) однозначно определяет идентификатор офицера дня. Сами описания офицеров (код, название, благоприятные и неблагоприятные действия) хранятся в `spr_indicator_value` для индикатора `day_officer`.

### 2.8. Справочник качества дня Мастера Дано

#### Таблица `spr_master_dano_mapping`

Матрица соответствия земной ветви месяца, небесного ствола дня и земной ветви дня качеству дня по методологии Мастера Дано.

- `month_branch_id` INTEGER FK → `spr_earthly_branch` — ветвь месяца;
- `day_stem_id` INTEGER FK → `spr_heavenly_stem` — ствол дня;
- `day_branch_id` INTEGER FK → `spr_earthly_branch` — ветвь дня;
- `indicator_value_id` INTEGER FK → `spr_indicator_value` — ссылка на значение индикатора `master_dano_day_quality`.

Сами значения индикатора (`+2`, `+1`, `0`, `-1`, `-2`) хранятся в `spr_indicator_value` с заполненной колонкой `numeric_score` (дробные числа допускаются) и текстовой интерпретацией.

### 2.9. Временная сетка

#### Таблица `t_time_grid_hourly`

Логическая (может быть реализована как VIEW, генератор или материализованная таблица) сетка почасовых отметок.

- `dt_utc` TIMESTAMP — момент начала часа (UTC)
- `dt_gmt0` TIMESTAMP — dt в GMT+0 (на текущем этапе совпадает с UTC)
- `year` INTEGER — григорианский год
- `month` INTEGER
- `day` INTEGER
- `hour` INTEGER (0..23)
- `weekday` INTEGER (1=Пн … 7=Вс)

Используется как базовый каркас для построения итогового VIEW.

## 3. Целевой VIEW: `v_bazi_calendar_hourly`

### 3.1. Назначение

VIEW `v_bazi_calendar_hourly` выдаёт для каждой часовой отметки времени (и для особых случаев смены сезона внутри часа) полный набор календарной и бацзы‑информации:

- дата/время,
- день недели,
- текущий солнечный сезон,
- четыре столпа (год/месяц/день/час) и их разложение на стволы/ветви.

### 3.2. Структура VIEW

Минимальный набор колонок (соответствует постановке задачи в `bazci_01.md`):

1. `dt_gmt0` TIMESTAMP — дата и время (обычно начало часа или момент смены сезона).
2. `weekday_short_ru` TEXT — день недели, сокращённо: Пн, Вт, Ср, Чт, Пт, Сб, Вс.
3. `hour_name_ru` TEXT — название часа (по животному: Крыса, Бык, …).
4. `hour_branch_char` TEXT — земная ветвь часа (иероглиф, например 卯).
5. `solar_term_id` INTEGER — номер сезона (1..24).
6. `solar_term_name_ru` TEXT — наименование сезона (русское).
7. `year_pillar` TEXT — столп года, конкатенация ствола и ветви (например 乙巳).
8. `month_pillar` TEXT — столп месяца.
9. `day_pillar` TEXT — столп дня.
10. `hour_pillar` TEXT — столп часа.
11. `year_stem_char` TEXT — небесный ствол года.
12. `year_branch_char` TEXT — земная ветвь года.
13. `month_stem_char` TEXT — небесный ствол месяца.
14. `month_branch_char` TEXT — земная ветвь месяца.
15. `day_stem_char` TEXT — небесный ствол дня.
16. `day_branch_char` TEXT — земная ветвь дня.
17. `hour_stem_char` TEXT — небесный ствол часа.
18. `hour_branch_char_dup` TEXT — земная ветвь часа (дублирование колонки 4 для единообразия схемы).
19. `lunar_month` INTEGER — номер лунного месяца китайского календаря (1–12, либо 0 при отсутствии данных).
20. `lunar_day` INTEGER — номер лунного дня (1–30, либо 0 при отсутствии данных).
21. `lunar_is_leap` INTEGER — флаг високосного месяца (1 — високосный, 0 — обычный).
22. `day_officer_value_id` INTEGER — ссылка на перечислимое значение показателя «12 офицеров».
23. `master_dano_value_id` INTEGER — ссылка на значение индикатора «качество дня Мастера Дано».
24. `master_dano_score` REAL — числовая интерпретация качества дня (если не заполнена, берётся `numeric_score` из справочника).

Дополнительно (опционально для внутренних join‑ов):
- `year_stem_id`, `year_branch_id`, `month_stem_id`, … — числовые идентификаторы стволов и ветвей.

## 4. Алгоритмы расчёта

### 4.1. Время и часовые пояса

- На текущем этапе **все расчёты выполняются в GMT+0 (UTC)**:
  - входное время либо уже задаётся в GMT+0,
  - либо внешний слой приводит локальное время пользователя к GMT+0.
- Астрономический расчёт моментов прохождения Солнцем заданных долгот выполняется в UTC (совпадает с GMT+0).
- Для расчёта земной ветви часа используется историческая таблица интервалов в GMT+8, поэтому:
  - время для определения ветви часа сначала переводится в GMT+8:
    - `t_gmt8 = t_utc + 8 часов`;
  - затем по `t_gmt8` определяется ветвь часа через `spr_hour_branch`.

### 4.2. Эталонные данные Гонконгской обсерватории

- Для лет 2025–2027 используется таблица `t_solar_term_time_hko`, которая инициализируется из официальных значений HKO (локальное время HKT = UTC+8).
- При запросе момента сезона модуль сначала ищет запись в `t_solar_term_time_hko`; если её нет, используется значение из `t_solar_term_time`.
- Пополнение делается скриптом `normalize_control.py`, который читает `Metodology/bazci_control_02.md`, конвертирует время в UTC и записывает в обе таблицы.

### 4.3. Опорные константы

- Опорный год `Y0` (например, 1864 или 1984) с известным столпом года `甲子`.
- 60‑летний цикл Дзя‑Цзы представлен в таблице `spr_pillar_cycle`.
- Опорный день `D0` (дата в григорианском календаре) с известным столпом дня (берётся из методологии/контроля) — используется для расчёта столпа дня.

### 4.3. Столп года

1. Для заданного `dt_gmt0` определяется предварительный григорианский год `Y`.
2. По таблице `t_solar_term_time` для года `Y` находится момент `crossing_gmt0` для сезона `Ли Чунь` (долгота 315°).
3. **Определение года бацзы:**
   - если `dt_gmt0 < crossing_gmt0(315°, Y)` → бацзы‑год = `Y - 1`;
   - иначе → бацзы‑год = `Y`.
4. Пусть бацзы‑год = `Y_bz`. Выполняется:
   - `delta_year = Y_bz - Y0`;
   - `cycle_index = (delta_year mod 60 + 60) mod 60` (всегда 0..59).
5. По `spr_pillar_cycle` с ключом `cycle_index` извлекаем `(stem_id, branch_id)` для годового столпа.

### 4.4. Солнечный сезон и солнечный месяц

1. Для каждого года в `t_solar_term_time` заданы все 24 записи (по `solar_term_id`).
2. Для заданного `dt_gmt0` находим последнюю по времени запись `crossing_gmt0 <= dt_gmt0`:
   - это текущий солнечный сезон `solar_term_id`.
3. Солнечный месяц определяется по номеру сезона:
   - `month_index = ceil(solar_term_id / 2)` (1..12).
4. Из `spr_solar_term` по `solar_term_id` берётся `month_branch_id` — земная ветвь месяца.

### 4.5. Столп месяца

1. Имеем:
   - небесный ствол года `year_stem_id` (из столпа года),
   - индекс солнечного месяца `month_index` (из п. 4.4).
2. По справочнику `spr_pillar_month_rule` выполняется поиск строки:
   - (`year_stem_id`, `month_index`) → `month_stem_id`.
3. Столп месяца:
   - ствол = `month_stem_id`, ветвь = `month_branch_id` (из `spr_solar_term`).

### 4.6. Столп дня

1. Выбирается опорная дата `D0` с известным столпом дня (по методологии или `bazci_control`).
2. Для любой даты `date` (часть `dt_gmt0` без времени) вычисляется количество дней от опорной даты:
   - `delta_days = (date - D0)` в целых днях.
3. Определяется индекс в 60‑летнем цикле:
   - `cycle_index = (delta_days mod 60 + 60) mod 60`.
4. По таблице `spr_pillar_cycle` с ключом `cycle_index` извлекается `(stem_id, branch_id)` для столпа дня.
5. **Граница дня:**
   - день начинается в `00:00` по времени GMT+0.

### 4.7. Земная ветвь часа

1. Для заданного `dt_gmt0` вычисляется время в GMT+8:
   - `t_gmt8 = dt_gmt0 + 8 часов`.
2. Из `t_gmt8` берётся час `H8` (0..23).
3. По таблице `spr_hour_branch` выбирается строка, для которой интервал `[start_hour_gmt8, end_hour_gmt8)` содержит `H8` с учётом переходов через полночь:
   - если `start_hour_gmt8 < end_hour_gmt8` → обычный интервал;
   - если `start_hour_gmt8 > end_hour_gmt8` → интервал «через 24:00»:
     - `H8 >= start_hour_gmt8` или `H8 < end_hour_gmt8`.
4. В найденной строке ветвь часа задаётся полем `branch_id` (и `branch_char`).

### 4.8. Небесный ствол часа

1. Имеем небесный ствол дня `day_stem_id` (из столпа дня).
2. Имеем земную ветвь часа `hour_branch_id` (из п. 4.7).
3. По таблице `spr_pillar_hour_rule` выполняется поиск строки:
   - (`day_stem_id`, `hour_branch_id`) → `hour_stem_id`.
4. Столп часа: ствол = `hour_stem_id`, ветвь = `hour_branch_id`.

### 4.9. Правило смены сезона внутри часа

1. Для часового интервала `[dt, dt+1h)` рассматриваем все записи `crossing_gmt0` из `t_solar_term_time`, попадающие строго внутрь интервала:
   - `dt < crossing_gmt0 < dt + 1h`.
2. Если ни одного пересечения нет:
   - для всех расчётов внутри часа используется один сезон (как в п. 4.4 по `dt`).
3. Если есть одно пересечение (что является типичным случаем):
   - создаются **две строки в VIEW**:
     1. Строка с сезонными полями, соответствующими **старому сезону**, с `dt_gmt0`, равным началу часа `dt`.
     2. Строка с сезонными полями, соответствующими **новому сезону**, с `dt_gmt0`, равным точному моменту `crossing_gmt0`.
4. Если астрономическое время смены сезона **совпадает с границей часа** (ровно в `dt` или `dt+1h`):
   - в соответствии с методологией, момент смены сезона относится уже к **новому сезону**;
   - это значит:
     - если `crossing_gmt0 == dt` → первый час целиком относится к новому сезону;
     - если `crossing_gmt0 == dt+1h` → текущий час целиком относится к старому сезону (пересечение произойдёт на следующем часе).

Точность момента смены сезонов — **до минут** (из `bazci_01_answers_01.md`).

## 5. Использование контрольных значений `bazci_control.md`

Файл `bazci_control.md` содержит подробную таблицу по 2025 и 2026 годам:

- для каждой двухчасовой зоны (тип Chen) указаны:
  - небесный ствол и земная ветвь года;
  - небесный ствол и земная ветвь месяца;
  - небесный ствол и земная ветвь дня;
  - небесный ствол и земная ветвь часа.

### 5.1. Загрузка в DuckDB

Планируется загрузить данные из `bazci_control.md` в таблицу, условно `t_bazi_control`:

- `date` — дата (формат `DD.MM.YY` или преобразованный в `DATE`),
- `time_range` — текстовое описание интервала (например, `23:00 (пред.дня) - 1:00`),
- `year_stem_char`, `year_branch_char`,
- `month_stem_char`, `month_branch_char`,
- `day_stem_char`, `day_branch_char`,
- `hour_stem_char`, `hour_branch_char`.

Дополнительно в процессе импорта можно вычислить момент начала интервала (`dt_start_gmt8`) и привести его к GMT+0.

### 5.2. Проверка через VIEW

Для верификации алгоритмов планируется запрос вида:

- сопоставить строки `t_bazi_control` со строками `v_bazi_calendar_hourly` по моменту начала интервала (после нормализации времени к GMT+0);
- сравнить все столпы и их составляющие:
  - `year_stem_char`, `year_branch_char`, …, `hour_stem_char`, `hour_branch_char`.

Любые расхождения фиксируются и используются для корректировки алгоритмов (опорного года/дня, таблиц правил и т.п.).
