# Производительность и Оптимизация (Performance & Optimization)

**Последнее обновление:** 15.02.2026

## 1. Принципы (Performance-First)

1.  **SQL-First:** Все массовые операции (JOIN, фильтрация, агрегация) выполняются в БД. Python — только оркестрация.
2.  **Иерархичность:** Не вычислять на уровне часа то, что не меняется в течение дня/месяца/года.
3.  **Batch-Processing:** Вставка и чтение данных только пакетами (по 10k-50k записей).
4.  **No-Loop:** Избегать циклов `for row in cursor` в Python для обработки данных.

## 2. Стратегия Иерархического Анализа

Для устранения дублирования вычислений и данных принята схема **Cascading Analysis**:

| Уровень | Частота (на 1 год) | Что анализируем | Куда пишем |
| :--- | :--- | :--- | :--- |
| **Год** | 1 | Столп года, годовые звезды, Найинь | `t_analysis_year` |
| **Месяц** | 12 | Столп месяца, Год-Месяц | `t_analysis_month` |
| **День** | ~365 | Столп дня, День-Месяц, День-Год, Офицеры, Донг | `t_analysis_day` |
| **Час** | ~8760 | Столп часа, Час-День/Месяц/Год, Расклады | `t_analysis_hour` |

### Преимущества
- **Storage:** Сокращение записей для дневных правил в 24 раза (365 vs 8760).
- **CPU:** Арифметическое сокращение количества проверок.
- **Maintenance:** Логическое разделение правил упрощает отладку.

## 3. Технические решения

### А. Оптимизация Базы Данных
- **Индексы:**
    - Обязательный индекс по `date_val` для `t_analysis_day`.
    - Обязательный composite index `(year, month)` для `t_analysis_month`.
    - Использование `BRIN` индексов для временных рядов (PostgreSQL) где применимо.
- **Partitioning:**
    - Таблицы часовых данных (`t_bazi_hourly`, `t_analysis_hour`) партицируются по годам при превышении 10 млн строк.

### Б. Алгоритмы
- **Шаблонизация:** Для Ци Мэнь используется поиск по 1080 шаблонам (`spr_qimen_templates`) вместо вычисления на лету.
- **Векторизация:** Для сложных Python-расчетов (если неизбежны) использовать матрицы, а не объекты.
