# Стратегия оптимизации: SQL-First Standard

**Статус:** Утвержденный стандарт разработки (с 15.02.2026)

## 1. Ключевой принцип: SQL-First

Любая массовая операция обработки данных (расчет календаря, анализ структур, поиск шаблонов) **должна** быть реализована на стороне базы данных (SQL), если это технически возможно.

Использование Python для итеративной обработки строк (`for row in cursor: ...`) допускается **только** в исключительных случаях, когда логику невозможно или нецелесообразно выразить декларативным SQL-запросом.

### Обоснование
Сравнительный анализ на модуле Тай И Шен Шу (февраль 2026) показал:
- **Python-цикл:** 5.60 сек.
- **SQL-запрос:** 0.19 сек.
- **Ускорение:** ~29 раз.

## 2. Критерии выбора реализации

### A. Когда использовать SQL (Native / CTE)
Используем SQL, если задача сводится к:
1.  **Множественным операциям:** Фильтрация, JOIN, агрегация.
2.  **Арифметике:** Вычисления по модулю, диапазоны, даты.
3.  **Поиску по шаблону:** Сравнение с эталонными таблицами (JSON extract, JOIN).
4.  **Генерации рядов:** `generate_series`, рекурсивные CTE.

**Примеры:**
- Расчет календаря (Тай И, Ци Мэнь).
- Проверка структур (Ворота + Звезда + Ствол).
- Поиск комбинаций стволов/ветвей.

### B. Когда использовать Python
Используем Python, если:
1.  **Сложная процедурная логика:** Требуется сохранение сложного состояния между шагами, которое трудно выразить в CTE.
2.  **Внешние библиотеки:** Нужна астрономия (`astropy`), сложные математические модели, отсутствующие в БД.
3.  **Динамический SQL:** Логика требует генерации запросов, структура которых неизвестна заранее.

## 3. Протокол разработки новых модулей/правил

1.  **Этап анализа:**
    -   Перед написанием кода задать вопрос: *"Можно ли это сделать одним SQL-запросом (INSERT INTO ... SELECT)?"*
    -   Если ответ "Да" — писать SQL.
    -   Если ответ "Возможно" — провести быстрый PoC (Proof of Concept) на SQL.
    
2.  **Этап реализации:**
    -   Избегать выборки данных в память приложения (`fetchall`) для последующей обработки.
    -   Использовать `INSERT INTO ... SELECT ...` для сохранения результатов.
    -   Использовать JSON-функции БД для работы со сложными структурами параметров.

3.  **Code Review:**
    -   Любой Pull Request/изменение, вводящее цикл обработки данных на Python, должно содержать обоснование, почему это нельзя сделать на SQL.

4.  **Классификация по времени (Hierarchical Check):**
    -   Определить частоту изменения параметра правила.
    -   Если параметр меняется 1 раз в день -> присвоить тип `Day` (не `Hour`!).
    -   **Строго запрещено** реализовывать дневные/месячные правила через часовые переборы (создает нагрузку x12-x24).

## Архитектура реализации (Hybrid Approach)

Несмотря на SQL-First, архитектура остается гибридной, где Python выступает в роли оркестратора.

### 1. SQL-Native Rules (Massive Set Operations)
- Правила, логика которых полностью укладывается в реляционные операции.
- **Реализация:** Генерация одного или нескольких сложных SQL-запросов.
- **Пример:** `CHECK_QIMEN_STEMS` (Структуры).
     ```sql
     INSERT INTO t_walk_analyze (...)
     SELECT ...
     FROM t_qumen_dgiren_hourly t
     JOIN spr_qimen_templates tmpl ON t.rasklad_id = tmpl.rasklad_id
     JOIN spr_rule_registry r ON r.predicate_code = 'CHECK_QIMEN_STEMS'
     WHERE tmpl.heaven_stem = json_extract(r.params_json, '$.heaven')
     ```

### 2. Python-Logic Rules (Complex/Procedural)
- Правила, требующие сложных вычислений или внешних библиотек.
- **Реализация:** Load -> Process -> Insert.
- **Пример:** Астрономические вычисления точного времени смены сезонов.

## 4. Стратегия Иерархического Анализа (Cascading Analysis)

**Концепция:**
Вместо плоского анализа каждого часа (Hour-Level), внедряется разделение по временным масштабам. Атрибуты высшего порядка (Год, Месяц, День) вычисляются один раз и наследуются часами.

### Проблема текущего подхода
Сейчас для года (8760 часов) правило "Год сталкивается с Месяцем" проверяется и записывается 8760 раз. Это создает избыточную нагрузку на CPU и раздувает объем БД (дублирование данных).

### Новая архитектура (To-Be)
1.  **Уровень Года (`t_analysis_year`):**
    -   *Частота:* 1 запись на год.
    -   *Проверки:* Столп Года, Звезды Года, Найинь Года.
    -   *Пример:* `2026`.
2.  **Уровень Месяца (`t_analysis_month`):**
    -   *Частота:* 12 записей на год.
    -   *Проверки:* Столп Месяца, Взаимодействие Год-Месяц, Звезды Месяца.
    -   *Пример:* `2026-02`.
3.  **Уровень Дня (`t_analysis_day`):**
    -   *Частота:* ~365 записей на год.
    -   *Проверки:* Столп Дня, Взаимодействие День-Месяц, День-Год, 12 Офицеров, Мастер Донг.
    -   *Пример:* `2026-02-15`.
4.  **Уровень Часа (`t_analysis_hour`):**
    -   *Частота:* 8760 записей на год.
    -   *Проверки:* Столп Часа, Взаимодействие Час-День/Месяц/Год.
    -   *Пример:* `2026-02-15 11:00`.

### Ожидаемый эффект
-   **Сокращение объема данных:** Результаты анализа года/месяца/дня хранятся в единственном экземпляре.
-   **Ускорение расчета:** Правила уровня дня вычисляются 365 раз вместо 8760 (ускорение в 24 раза для этого слоя).
