# Анализ производительности и структуры проекта Calk_KMF

## 1. Текущее состояние базы данных

На основе анализа файла `calk_kmf.sqlite`:

- **Основная таблица `t_bazi_hourly`**: 237,830 записей.
    - Содержит рассчитанные столпы для каждого часа в каждом часовом поясе.
    - **Проблема**: Отсутствовал индекс по полям UTC, используемым в основном представлении `v_bazi_hourly` (фильтр `slot_start_date_utc >= date('now')`).
    - **Решение**: Добавлен индекс `idx_bazi_hourly_utc` (tz_offset_hours, slot_start_date_utc, slot_start_time_utc).

- **Таблицы аналитики**:
    - `t_analyz_data` (Tong Shu): 2,027 записей. Структура EAV (Entity-Attribute-Value).
    - `t_qumen_chauby` (Qi Men): 9,000 записей. Нормализованная структура.

- **Справочники**: Заполнены корректно, объем данных небольшой.

## 2. Анализ кода и алгоритмов

### 2.1. Расчет столпов (`code/bazi_calendar/hourly.py`)

**Выявленные проблемы:**
1.  **Избыточность вычислений**: Функция `populate_bazi_hourly_for_years` перебирает все смещения часовых поясов (от -12 до +14, итого ~27 проходов). Для каждого пояса вызывается `calc_four_pillars`, который выполняет астрономические вычисления (координаты солнца, лунные фазы).
    - *Суть*: Положение планет зависит от момента времени UTC. Вычислять их заново для каждого часового пояса избыточно. Различается только локальное время и (иногда) границы дня.
2.  **Обработка данных**: Вставка происходит батчами (`executemany`), что хорошо. Но весь объем данных генерируется в Python.

**Рекомендации по оптимизации:**
- **Кэширование астрономии**: Реализовать кэш для результатов `calc_four_pillars` на основе `slot_start_date_utc`. Если UTC время совпадает, астрономические данные (Solar Term, Lunar Phase) идентичны. Меняются только `Day Pillar` (сдвиг начала суток) и `Hour Pillar`.
- **Векторизация**: Использование `numpy` или `pandas` для массового расчета дат могло бы ускорить процесс на порядок, но требует добавления тяжелых зависимостей. Для текущего объема (сотни тысяч строк) чистый Python пока справляется.

### 2.2. Расчет Тун Шу (`code/bazi_calendar/tongshu/engine.py`)

**Выявленные проблемы:**
- **Полный пересчет**: Метод `populate_analysis_table` очищает таблицу и считает все заново.
- **Цикл в Python**: Обработка идет построчно.

**Рекомендации:**
- **Инкрементальное обновление**: Добавить возможность пересчитывать только новые или изменившиеся часы, вместо полной очистки.

### 2.3. Структура данных

**Таблица `t_bazi_hourly`:**
- Поле `solar_term_name` (TEXT) дублирует данные из `spr_solar_term`. Предлагается удалить его из таблицы, оставив только `solar_term_id`, так как имя подтягивается через JOIN в представлении. Это сэкономит место.
- Хранение столпов в виде строк (`year_pillar`, `year_stem`...) удобно для чтения, но избыточно. Хранение ID (1-60) было бы компактнее, но усложнило бы отладку. Текущий подход приемлем для SQLite.

**Таблица `t_analyz_data`:**
- Структура EAV (`indicator_code`, `value`) гибкая, но медленная для аналитических выборок (Pivot).
- Если набор показателей стабилизируется, рекомендуется "расплющить" таблицу, выделив частые показатели (например, 12 офицеров, На Инь) в отдельные колонки таблицы фактов.

## 3. Этапность расчетов

Текущая цепочка:
1.  `populate_spr.py` -> Загрузка справочников.
2.  `run_update.py` -> Расчет `t_bazi_hourly` (тяжелая операция).
3.  `run_tongshu_update.py` -> Расчет аналитики на основе `t_bazi_hourly`.
4.  `qimen/run_update.py` -> Расчет Ци Мэнь.

Цепочка логична. Главное узкое место — шаг 2.

### 5. Оптимизация Модуля Анализа (Rule Engine)

**Выявленная проблема:** Построчная проверка 100+ правил для 1M+ записей в Python занимала минуты/часы (O(N*M)).

**Реализованные решения:**
1.  **Lookup Map (Кэширование правил):** Вместо перебора всех правил, для каждой строки формируется ключ (например, `НебесныйСтвол + ЗемнойСтвол`), и правила выбираются из хэш-таблицы за O(1).
    - *Результат:* Расчет 1 млн строк за 50 секунд.
2.  **Оптимизация записи в SQLite:** Использование `PRAGMA synchronous = OFF` и `journal_mode = MEMORY` для массовых вставок ускоряет запись в 5-10 раз.

**Стратегическое решение (Шаблонизация):**
- Для систем с конечным числом состояний (Ци Мэнь = 1080 уникальных карт) вычисления заменяются на **Поиск Шаблона**.
- Создана таблица `spr_qimen_templates`. Вместо вычисления структуры каждой карты "на лету", скрипт будет находить ID шаблона (`YANG_1_JIA_ZI`) и брать готовые данные.
