# 2026-03-09: Исправление дневного расклада Ци Мэнь Джи Рэн + удаление Чай Бу day/month/year

## Статус
**УСПЕХ**. Исправлена формула расчёта Yuan (level) для дневного расклада Ци Мэнь Джи Рэн. Удалены таблицы и код Чай Бу для day/month/year (оставлен только hourly).

## Проблема
Формула расчёта Yuan (Upper/Middle/Lower) для дневного расклада давала неверные Ju-номера. Старая формула `(((abs_cycle - term_cycle) % 3) + 1)` использовала разницу 60-дневных циклов между текущей датой и началом солнечного термина — это давало нестабильные результаты.

## Найденная формула
Yuan level определяется **позицией солнечного термина** в его полугодии (Инь или Ян):
- **Yang полугодие** (term 22→pos1, 23→2, 24→3, 1→4, 2→5, ..., 9→12)
- **Yin полугодие** (term 10→pos1, 11→2, ..., 21→12)
- **Level** = `((pos - 1) // 4) + 1`: позиции 1-4 = Upper(1), 5-8 = Middle(2), 9-12 = Lower(3)

## Верификация
8/8 контрольных дат 2026 года — 100% совпадение:
| Дата       | Ожидание | term | pos | Level  | Ju | OK |
|------------|----------|------|-----|--------|----|----|
| 2026-03-01 | Ян 8     | 2    | 5   | Middle | 8  | ✓  |
| 2026-03-05 | Ян 3     | 3    | 6   | Middle | 3  | ✓  |
| 2026-03-06 | Ян 4     | 3    | 6   | Middle | 4  | ✓  |
| 2026-03-24 | Ян 4     | 4    | 7   | Middle | 4  | ✓  |
| 2026-04-16 | Ян 9     | 5    | 8   | Middle | 9  | ✓  |
| 2026-04-30 | Ян 5     | 6    | 9   | Lower  | 5  | ✓  |
| 2026-09-17 | Инь 9    | 15   | 6   | Middle | 9  | ✓  |
| 2026-12-18 | Инь 4    | 21   | 12  | Lower  | 4  | ✓  |

## Изменённые файлы

# 2026-02-28: Модуль синхронизации событий с Яндекс Календарём

## Статус
**УСПЕХ**. Создан модуль `code/calendar/yandex_sync.py` для автоматической генерации событий из Бацзы-данных и синхронизации с Яндекс Календарём по протоколу CalDAV.

## Выполненные задачи

### 1. Таблица t_event
- **t_event** (PK: id TEXT = MD5-хэш): dt_start, dt_end (timestamp), calendar, title, comment
- ID = md5(dt_start::text || dt_end::text || calendar || title || COALESCE(comment, ''))
- События на весь день: dt_start = 00:00, dt_end = 00:00 следующего дня

### 2. Автоматическое наполнение из Бацзы (SQL-First, один запрос с UNION ALL)
- **День Устранения** — офицер дня = Устранение
- **День Наполнения** — офицер дня = Наполнение
- **Мытье двери** — офицер = Успех / Сбор урожая / Открытие
- **Грабитель богатства** — НС дня = 癸 (личный)
- **Благородный** — ЗВ дня = 巳 или 卯 (личный)
- Всего: ~753 события (все годы в t_bazi_hourly)

### 3. CalDAV-синхронизация
- Библиотека: `caldav` (PyPI) + `vobject` для парсинга iCalendar
- Целевой календарь: "Ключевые события"
- Алгоритм: сравнение UID (= t_event.id) между БД и Яндекс Календарём
- Идемпотентность подтверждена: повторный запуск → kept=20, created=0, deleted=0

### 4. Порядок выполнения модуля
1. Создание t_event (если нужно)
2. Наполнение из Бацзы (DELETE + INSERT, полная перезапись)
3. Удаление прошедших событий (dt_end < now МСК)
4. Пересчёт ID (writable CTE: DELETE RETURNING + INSERT)
5. Синхронизация с Яндекс Календарём (окно: now … now + 30 дней)

# 2026-02-18: Исправление алгоритма Черного Кролика v2 (Joey Yap) — Полная реконструкция

## Статус
**УСПЕХ**. Алгоритм полностью пересмотрен и верифицирован на 7 новых контрольных датах 2026 года.

## Проблема
Предыдущий алгоритм (v1, "ZigZag Ring") давал расхождения на новых контрольных датах 2026-2027.
Brute-force анализ показал, что Palace Walk из методологии НЕ может дать правильный результат для всех дат.
Ключевое открытие: стартовый дворец 1-го ЛД определяется lookup-таблицей по `(first_day_jiazi_id - 1) % 9`.

## Реализованный Алгоритм v2 (Reverse Engineered)
1.  **Вычислить `jiazi_id` 1-го лунного дня:** `fjz = ((day_jiazi_id - lunar_day) % 60 + 60) % 60 + 1`.
2.  **Стартовый дворец:** `first_palace = LOOKUP[(fjz - 1) % 9]`.
    *   `LOOKUP = [1, 2, 6, 3, 4, 2, 1, 3, 6]`.
    *   Позиции 0–6 подтверждены данными; позиции 7, 8 — предсказаны по паттерну чередования.
3.  **Направление:** **Стандартная** классификация Инь/Ян ствола 1-го ЛД.
    *   Чётный stem_idx (甲丙戊庚壬) = Yang → CW: `[1, 8, 3, 4, 9, 2, 7, 6, 5]`.
    *   Нечётный stem_idx (乙丁己辛癸) = Yin → CCW: `[1, 6, 7, 2, 9, 4, 3, 8, 5]`.
4.  **Дворец текущего дня:** `seq[(seq.index(first_palace) + lunar_day - 1) % 9]`.

## Ключевые отличия от v1
-   **Убран Palace Walk** (поиск кролика → хождение по дворцам). Заменён на LOOKUP-таблицу.
-   **Стандартная Инь/Ян** вместо специальной классификации из методологии для дневного метода.
-   Часовой метод (`calculate_hour_star`) оставлен без изменений — использует специальную классификацию и Таблицу 4.

# 2026-02-15: Исправление данных и алгоритмов Анализа Года (Post-Mortem)

## Статус
**УСПЕХ**. Устранены ошибки в расчете правил года для 2026 года (Наинь, Фазы Ци).
Успешно обработана ситуация смена столпа года (Ли Чунь): в 2026 году присутствуют и 'Бин У' и 'И Сы'.

## Проблема
Пользователь обнаружил некорректные данные для 2026 года:
1.  **Фаза Ци:** Выводилось "Year Rule" вместо "Императорский светильник".
2.  **Наинь:** Выводился "Огонь светильника" вместо "Вода млечного пути".
3.  **Конфликты PK:** При наличии двух столпов года (до и после 4 февраля) возникал конфликт первичного ключа в `t_analysis_year`.

## Анализ причин (Root Cause)
1.  **Данные:** В справочнике `spr_jiazi_extended` для столпа Бин У (43) были устаревшие данные (Огонь). В справочнике `spr_bazi_qi_phase` у фазы 5 было битое имя.
2.  **Алгоритм:** В `run_analysis.py` отсутствовала специфическая ветка для правила `CHECK_QI_PHASE_PILLAR_YEAR`, из-за чего срабатывал `ELSE 'Year Rule'`.
3.  **Архитектура:** Таблица `t_analysis_year` имела первичный ключ `(year, rule_id)`, что не позволяло хранить два значения для одного года (смена года происходит 4 февраля, поэтому в 2026 году присутствуют и данные 2025-го).

## Решение
1.  **Исправление данных:** SQL-запросы для обновления справочников `spr_jiazi_extended` и `spr_bazi_qi_phase`.
2.  **Обновление схемы:** 
    - `t_analysis_year` пересоздана с PK `(year, rule_id, result_value, year_pillar)`.
    - `t_analysis_month` пересоздана с PK `(year, month, rule_id, result_value, year_pillar, month_pillar)`.
    - Аналогично обновлены PK для остальных таблиц анализа, включая направления, чтобы уникально идентифицировать результаты для разных наборов столпов.
3.  **Обновление кода:** 
    - Добавлена логика расчета Фаз Ци года с подтягиванием имен и баллов.
    - Обновлены SQL запросы в `run_analysis.py` для заполнения колонок столпов (`year_pillar`, `month_pillar` и т.д.).
    - Исправлены `ON CONFLICT` выражения.

## Результаты верификации (2026)
- **Столпы года:** Обнаружены 'Бин У' и 'И Сы' (корректно для переходящего года).
- **Записи:**
  - `t_analysis_year`: 14 записей
  - `t_analysis_month`: 168 записей
  - `t_analysis_day`: 6032 записи
  - `t_analysis_direction_day`: 3016 записей

## Извлеченные уроки (Technical Lessons)
1.  **Кодировка Windows:** Скрипты диагностики падали с `UnicodeEncodeError`. **Решение:** Внедрить стандарт вывода в файл или `sys.stdout.reconfigure`.
2.  **Специфика Бацзы:** Год не атомарен. Любой агрегат "по году" должен учитывать возможность наличия двух разных столпов года в одном календарном году.
3.  **Драйвер БД:** `pg8000` возвращает кортежи, а не словари. Это замедляет написание ad-hoc скриптов. Требуется учитывать при отладке.
