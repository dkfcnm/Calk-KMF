# Спецификация универсального модуля анализа (Rule Engine)

## 1. Архитектура
Модуль построен по принципу **Database-Driven Engine**. Логика проверок (Предикаты) отделена от конфигурации правил (БД).

### Компоненты
1.  **БД (SQLite):** Хранит определения правил, настройки и результаты.
2.  **`code/analysis/engine.py`:** Движок, загружающий правила и запускающий проверки.
3.  **`code/analysis/predicates.py`:** Библиотека функций-проверок (Python-код).

## 2. Схема Базы Данных (Детально)

### Таблица `t_rule_registry` (Реестр правил)
Хранит определение логики правила и его базовую оценку.
*   **`rule_id` (TEXT, PK):** Уникальный идентификатор правила. Генерируется автоматически как MD5-хэш от параметров логики (`params_json + score_formula`).
*   **`period_type` (TEXT):** Тип периода расчета (`year`, `month`, `day`, `hour`). Определяет, на каком уровне иерархии выполняется правило.
*   **`name_ru` (TEXT):** Понятное название правила на русском языке (например, "Дерево на Огне").
*   **`predicate_code` (TEXT):** Ссылка на функцию-предикат в файле `code/analysis/predicates.py` (например, `CHECK_QIMEN_STEMS`).
*   **`params_json` (TEXT):** Параметры для функции проверки в формате JSON (например, `{"heaven": "甲", "earth": "丙"}`).
*   **`score_base` (REAL):** Числовая оценка. Положительное значение — благоприятно, отрицательное — неблагоприятно.
*   **`score_formula` (TEXT):** (Опционально) Формула для сложного расчета оценки (например, `base * 2`).
*   **`description` (TEXT):** Подробное описание методологии.
*   **`is_active` (INTEGER):** Флаг активности (`1` - включено, `0` - выключено).

### Таблица `t_rule_scope` (Области применения)
Определяет, в каких контекстах проверяется правило.
*   **`rule_id` (TEXT, PK, FK):** Ссылка на правило.
*   **`scope_type` (TEXT, PK):** Тип области:
    *   `direction`: Анализ направлений (расклады Ци Мэнь).
    *   `date`: Анализ характеристик даты (Бацзы, Тун Шу).
    *   `activation`: Поиск активаций.
*   **`is_stop` (INTEGER):** Флаг "Стоп-фактор". Если `1`, срабатывание правила прекращает дальнейший анализ в этой области или помечает результат как блокирующий.

### 2.1 Справочники Правил (Rule Dictionaries)
Для сложных методологий (Тун Шу, Бацзы) используются вспомогательные таблицы, чтобы не хардкодить логику в SQL.

#### `spr_tongshu_stem_combo_rule` / `spr_tongshu_branch_combo_rule`
Правила комбинаций Небесных Стволов и Земных Ветвей (Слияния, Столкновения, Наказания).
*   `combo_name`: Название комбинации.
*   `combo_type_id`: ID типа (1=Слияние, 2=Столкновение и т.д.).
*   `item1`, `item2`, `item3`: Участники комбинации.
*   `numeric_score`: Оценка.

#### `spr_bazi_qi_phase`
Таблица Фаз Ци (12 фаз).
*   `stem_char`: Небесный ствол.
*   `branch_char`: Земная ветвь.
*   `phase_name`: Название фазы (Рождение, Пик рассвета...).
*   `numeric_score`: Балл.

#### `spr_jiazi_extended`
Расширенные характеристики 60 столпов Цзя-Цзы.
*   `jiazi_id`: 1-60.
*   `nayin_element`, `nayin_name`: Наинь.
*   `dagua_element`, `dagua_period`, `dagua_family`, `dagua_role`: И-Цзин (Да Гуа).

#### `spr_black_rabbit_matrix` / `spr_black_rabbit_scores`
Система "Черный Кролик" (Звезды Солнца и Луны).
*   **Вариант Joey Yap (`spr_black_rabbit_day_joey`):**
    *   Алгоритм восстановлен (Reverse Engineering, v2, 2026-02-15) на основе 7 контрольных дат.
    *   **Структура таблицы:** 1800 записей (60 first_day_jiazi_id × 30 lunar_day).
    *   **Алгоритм расчёта дворца:**
        1.  Вычислить `jiazi_id` 1-го лунного дня: `fjz = ((day_jiazi_id - lunar_day) % 60 + 60) % 60 + 1`.
        2.  Стартовый дворец: `first_palace = LOOKUP[(fjz - 1) % 9]`.
            *   `LOOKUP = [1, 2, 6, 3, 4, 2, 1, 3, 6]` (позиции 0-6 подтверждены, 7-8 предсказаны).
        3.  Направление: **стандартная** Инь/Ян ствола 1-го ЛД (чётный индекс = Yang = CW, нечётный = Yin = CCW).
            *   CW: `[1, 8, 3, 4, 9, 2, 7, 6, 5]`
            *   CCW: `[1, 6, 7, 2, 9, 4, 3, 8, 5]`
        4.  Дворец текущего дня: `seq[(seq.index(first_palace) + lunar_day - 1) % 9]`.
    *   **Важно:** Дневной метод использует **стандартную** классификацию Инь/Ян (НЕ специальную из методологии). Часовой метод использует специальную классификацию и Таблицу 4.
    *   **Реализация:** `code/analysis/black_rabbit_logic.py` → `calculate_day_star()`.

#### `spr_yanqin_day_constellation`
Система "Цинь Птицы" 演禽 (Yan Qin) — 28 созвездий.
*   **Структура:** 28 строк (7 дней недели × 4 группы ветвей).
*   **Ключи:** `dow` (0=Вс...6=Сб) + `branch_group` (1=申子辰, 2=巳酉丑, 3=寅午戌, 4=亥卯未).
*   **Данные:** `constellation_char`, `constellation_name`, `star_element` (планета), `score` (+1/-1).
*   **Скоринг:** Традиционная классификация 通書 (14 吉 = +1, 14 凶 = -1).
*   **Правило:** `CHECK_YANQIN_DAY`, period_type=`day`.
*   **Реализация:** SQL JOIN в `run_analysis.py` → `process_day_level()`.

#### `spr_taiyi_jianchu`
12 созвездий 建除十二神 для системы Тай И Шен Шу.
*   **Структура:** 12 строк (建, 除, 满, 平, 定, 执, 破, 危, 成, 收, 开, 闭).
*   **Ключ:** `id` (1-12).
*   **Данные:** `char_cn` (иероглиф), `name_ru` (русское название), `description`, `score` (+1/0/-1).
*   **Расчёт:** По лунному месяцу. 建 стартует с ветви лунного месяца (месяц 1→寅, 2→卯, ...).

#### `t_qumen_tayi_day` (v2, плоский формат)
Итоговая таблица дневного расклада Тай И Шен Шу (21 колонка, 9 строк/день = 1 строка/дворец).
*   **PK:** (date_val, palace_no).
*   **Столпы:** year_pillar, month_pillar, day_pillar.
*   **Расклад:** run_type, tai_yi_palace, xiu_men_palace, xi_shen_palace.
*   **Дворец:** star (иероглиф), gate (иероглиф, NULL для P5-центр).
*   **Часовые данные по направлению:** hhd_spirits (ХХД иероглифы, "/" для 2-ветвевых), jianchu (建除 иероглифы).
*   **Флаги:** is_xi_shen, is_noble (OR), is_kong_wang (OR).
*   **Scores:** gate_score, star_score, jianchu_score (avg для 2-ветвевых).
*   **Формула:** total_score = (gate×2 + star×2 + jianchu + xi_shen + noble) × (1 − kong_wang).
*   **Правило дня:** `CHECK_TAIYI_DAY_QUALITY` — JOIN по `palace_no = tai_yi_palace`.
*   **Правило часа:** `CHECK_TAIYI_NOBLE_HOUR` — из t_taiyi_hours напрямую.

#### `spr_yellow_black_matrix` / `spr_yellow_black_stars`
Методология "Желтый и Черный путь".
*   Матрица: Ветвь Месяца + Ветвь Дня -> ID Звезды.
*   Звезды: ID, Название, Оценка (Желтый +1, Черный -1).

### 2.2 Форматы результатов правил

Некоторые правила возвращают сложные форматированные значения:

#### Фазы Ци (Qi Phases)
Правила `CHECK_QI_PHASE_PILLAR_YEAR` и `CHECK_QI_PHASE_PILLAR_HOUR` возвращают двойное значение (Фаза Столпа / Фаза Месяца).
Правило `CHECK_QI_PHASE_PILLAR_DAY` возвращает одиночное значение (только Фаза Столпа).

*   **Формат (Dual):** `Фаза_Столпа (НС:X - ЗВ:Y) / Фаза_Месяца (НС:M - ЗВ:Y)`
*   **Формат (Single):** `Фаза_Столпа (НС:X - ЗВ:Y)`
*   **Пример:** `Купание (НС:庚 - ЗВ:午) / Императорский светильник (НС:丙 - ЗВ:午)`

#### Лунные Даты (Lunar Dates)
Правила `CHECK_LUNAR_MONTH_NUM` и `CHECK_LUNAR_DAY_NUM` выводят порядковые номера по лунному календарю.
*   **Лунный месяц:** `X` (или `X (Високосный)`).
*   **Лунный день:** `Y`.

## 3. Руководство: Как управлять правилами

### Как добавить новое правило (вручную через БД)
1.  **Выберите Предикат:** Определите, какая логика нужна (например, проверка сочетания стволов). Узнайте код предиката (например, `CHECK_QIMEN_STEMS`) и требуемые параметры.
2.  **Подготовьте JSON:** Сформируйте строку параметров, например `{"heaven": "甲", "earth": "丙"}`. Используйте **Иероглифы**, а не транскрипцию.
3.  **Вставьте в `t_rule_registry`:**
    *   `rule_id`: Придумайте временный или сгенерируйте хэш (лучше использовать скрипты импорта).
    *   `name_ru`: Название.
    *   `predicate_code`: Код из шага 1.
    *   `params_json`: Строка из шага 2.
    *   `score_base`: Оценка (например, `1`).
4.  **Вставьте в `t_rule_scope`:**
    *   Добавьте запись с `rule_id` и `scope_type='direction'` (или другой).
    *   **ВНИМАНИЕ:** Никогда не добавляйте `scope_type='activation'`, если это явно не указано в методологии, даже если оценка положительная.

### Как изменить оценку правила
1.  Найдите правило в `t_rule_registry` по названию или ID.
2.  Измените значение в колонке `score_base`.
3.  Сохраните изменения. При следующем расчете движок подхватит новое значение.

### Как отключить правило
1.  Найдите правило в `t_rule_registry`.
2.  Установите `is_active = 0`.

### Импорт правил из файлов
Для массовой загрузки (например, 100 структур) используйте скрипты в папке `code/utils/`. Пример: `import_qimen_structures.py`. Скрипт автоматически:
*   Генерирует правильные `rule_id` (хэши).
*   Чистит старые версии правил.
*   Заполняет обе таблицы (`registry` и `scope`).

## 4. Протокол внесения нового правила (Чек-лист)

При получении методологии правила от пользователя, необходимо проверить наличие всех обязательных атрибутов и оценить способ реализации. **Если что-то отсутствует — задать вопрос.**

**Чек-лист:**
1.  **Оценка реализации (SQL vs Python) - ВАЖНО:**
    *   *Вопрос:* Можно ли реализовать проверку одним SQL-запросом (через JOIN, диапазоны, JSON_EXTRACT)?
    *   Если ДА -> Требовать/Планировать реализацию через SQL-Native Rule.
    *   Если НЕТ -> Обосновать необходимость использования Python-Logic.

2.  **Логика проверки (Предикат):**
    *   Что именно проверяем? (Столпы, Дворец, Звезды, Сезон...).
    *   Нужен новый предикат или подходит существующий?
    *   Какие параметры будут в JSON? (Гибкость: например, не "Крыса", а "Ветвь А").

3.  **Область распространения (Scope):**
    *   Анализ даты?
    *   Направление (расклад)?
    *   Активация?
    *   (Может быть несколько).

4.  **Оценка (Score):**
    *   Конкретное число? (например, -10).
    *   Формула?
    *   Положительная (хорошо) или отрицательная (плохо)?

5.  **Стоп-фактор (Stop Factor):**
    *   Является ли блокирующим? Для какой именно области?

6.  **Метаданные:**
    *   Уникальный ID (slug).
    *   Краткое название.
    *   Комментарий/Описание.

7.  **Флаг активности:** По умолчанию 1.

8.  **Временной масштаб (Period Type) - ОБЯЗАТЕЛЬНО:**
    *   Определить, как часто меняется значение правила:
        *   `year`: Раз в год (Найинь года, Звезды года).
        *   `month`: Раз в месяц (Столп месяца, Летящие звезды месяца).
        *   `day`: Раз в день (12 Офицеров, Мастер Донг, Звезды дня).
        *   `hour`: Раз в час (Расклад Ци Мэнь, Структуры).
    *   **Правило:** Если параметр не меняется в течение дня, запрещено ставить тип `hour`.

## 5. Технические стандарты (Kaizen)

### Кодировка и Вывод
Поскольку проект работает в среде Windows, любой Python-скрипт, выводящий данные в консоль (print), **обязан** содержать следующий блок инициализации для поддержки иероглифов:
```python
import sys
if sys.platform == "win32":
    try:
        sys.stdout.reconfigure(encoding='utf-8')
    except AttributeError:
        pass
```

### Форматы данных
*   **Небесные стволы / Земные ветви:** В базе данных и в правилах хранятся и используются **Иероглифы** (甲, 子), а не Пиньинь. Предикаты должны ожидать иероглифы.
*   **Проверка данных:** Перед написанием нового предиката необходимо выполнить `SELECT` запрос к целевой таблице (`t_qumen_...`, `t_bazi_...`), чтобы удостовериться в формате данных.

### Импорт правил
*   Для массовой загрузки правил (из Markdown, Excel) использовать скрипты-импортеры в папке `code/utils/`, а не SQL-инъекции вручную.
