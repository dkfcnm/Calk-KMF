# Технические стандарты и работа над ошибками (15.02.2026)

Этот документ фиксирует стандарты разработки скриптов и взаимодействия с окружением, выработанные на основе решения задач по интеграции модулей и управлению процессами.

## 1. Файловые операции и Shell
**Проблема:** Команды оболочки (`del`, `rm`, `cp`) ненадежны из-за различий в OS (Windows CMD vs PowerShell) и окружении.
**Решение (Стандарт):**
- **Всегда использовать Python** для манипуляций с файлами.
- Вместо `run_command("del file")` использовать:
  ```python
  import os
  if os.path.exists("file"):
      os.remove("file")
  ```
- Для массового удаления создавать временный скрипт очистки `scripts/cleanup_temp.py`.

## 2. Окружение Python и Импорты
**Проблема:** Вспомогательные скрипты в папке `scripts/` не видят модули проекта (`code/`), так как корень не в `PYTHONPATH`. Команда `python` может не работать.
**Решение (Стандарт):**
- **sys.path:** В начале любого скрипта в подпапке добавлять:
  ```python
  import sys, os
  # Добавляем корень проекта (на 2 уровня выше скрипта)
  project_root = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
  if project_root not in sys.path:
      sys.path.insert(0, project_root)
  ```
- **Запуск:** Использовать `py` (Windows Launcher) или определять путь к интерпретатору динамически.

## 3. Взаимодействие с БД (PostgreSQL)
**Проблема:** Длительные транзакции блокируют миграции. Хардкод паролей в скриптах проверки. Прямые подключения могут быть небезопасны или нестабильны в разных окружениях.
**Решение (Стандарт):**
- **MCP Сервер:** Все обращения к базе данных PostgreSQL (чтение, запись, аналитика) должны осуществляться **исключительно через MCP сервер** (Model Context Protocol). Прямые подключения из скриптов или ORM в обход MCP сервера запрещены.
- **Конфигурация:** Всегда загружать `.env` файл или использовать `code.common.db_config`. Не хардкодить учетные данные.
- **Управление процессами:**
  1. Проверять активность: `SELECT * FROM pg_stat_activity WHERE state != 'idle'`.
  2. Останавливать мягко: `pg_cancel_backend(pid)`.
  3. Останавливать жестко: `pg_terminate_backend(pid)` (только свои запросы).
- **Не прерывать autovacuum** без крайней необходимости.

## 4. UI-элементы и идентификация
**Проблема:** При коммуникации с AI-агентом сложно точно указать на конкретный элемент интерфейса.
**Решение (Стандарт):**
- **Каждый создаваемый React-компонент**, который является значимым элементом интерфейса (кнопка, поле, вкладка, ячейка, карточка), **обязан** получать атрибут `data-element-id` в формате `area:element:identifier`.
- Для оборачиваемых элементов использовать `ElementIdBadge` или `useElementId` hook.
- Для элементов, которые нельзя обернуть (TableRow, TableCell), добавлять `data-element-id` напрямую через `generateId()`.
- **В верхнем правом углу** каждого элемента с `data-element-id` отображается знак **¡ (U+00A1)** бледно-серым цветом. При нажатии на ¡ открывается popup с ID элемента, при нажатии на ID — копирование в буфер обмена.
- Отображение ¡ управляется из раздела «Администратор» и сохраняется в `localStorage` (`kmf_element_id_markers`).

## 5. Очистка после работы
**Проблема:** Накопление временных файлов `check_*.py`, `verify_*.py`.
**Решение (Стандарт):**
- В конце задачи всегда запускать проверку наличия временных скриптов и удалять их (через Python-скрипт).
