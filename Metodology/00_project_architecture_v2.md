# Архитектура проекта Calk_KMF (v2.3)

**Последнее обновление: 31.05.2026**

## Общий подход
- **Rolling Window Strategy (3 Years):** Анализ и расчеты выполняются и хранятся для скользящего окна в 3 года (текущий: **2025, 2026, 2027**). Это обеспечивает мгновенный отклик интерфейса без необходимости вычислений на лету.
- **SQL-First Strategy:** Все массовые расчеты (календарь, структуры, поиск шаблонов) выполняются средствами SQL (CTE, Window Functions, JOIN) внутри базы данных. Python используется только как оркестратор и для логики, нереализуемой в SQL.
- **Хранение в PostgreSQL (опорная БД):** Основная база данных проекта — PostgreSQL (`calk_kmf`). SQLite выступает вспомогательной (fallback) базой для агрегатных представлений и ускорения работы портала в offline-режиме.
- **MCP интеграция:** Настроен MCP сервер (`@modelcontextprotocol/server-postgres`) для работы с БД из IDE. Все DDL операции рекомендуется выполнять через MCP.
- **Отсутствие хардкода:** Константы (стволы, ветви, таблицы Ju, карты звезд) вынесены в таблицы `spr_...` и загружаются при старте через `code.common.config`.
- **Высокая производительность:** Используются оптимизированные алгоритмы, шаблонные таблицы, пакетная обработка данных.
- **Обязательные комментарии к DDL:** При создании или изменении любой таблицы/колонки ОБЯЗАТЕЛЬНО добавлять `COMMENT ON TABLE` и `COMMENT ON COLUMN`. DDL-экспорт комментариев хранится в `data/ddl_comments.sql`.

## Важные архитектурные решения

1. **SQL-First Standard:** Приоритет реализации логики на SQL. Python используется только если SQL реализация невозможна или неэффективна (пример: сложная астрономия).
2. **Шаблоны Ци Мени:** Вместо вычисления на лету используется поиск по 1080 шаблонам (`spr_qimen_templates`).
3. **Гибридный анализ:** SQL для структурных правил, Python для сложной процедурной логики.
4. **WAL режим:** База работает в WAL режиме для параллельных чтений.
5. **Пакетная обработка:** Все массовые операции используют батчи по 50K записей.
6. **Валидация данных:** Контрольные таблицы для регрессионного тестирования.
7. **Резервное копирование:** Используется `pg_dump` (Custom format) для PostgreSQL и копирование файла для SQLite.
8. **Classical Qimen Method:** Часовые расклады Чжи Рэн рассчитываются по классической методологии (fu_tou от day_pillar, solar_term, yuan_idx). `curr_date` для hourly использует LOCAL дату (не UTC), чтобы учесть смену дня в 子 (23:00).

## Правила рефакторинга базы данных

При изменении структуры таблиц:
1. Если таблица данных переименовывается, обязательно переименовать или создать соответствующую контрольную таблицу.
2. Обновить `code/tests/control_data.py` (словарь `TABLES_PK`).
3. Использовать MCP сервер (`query_db`) для DDL операций вместо создания временных скриптов.
4. После изменений обязательно запустить регрессионные тесты.
