# ПРЕДЛОЖЕНИЯ (П)

## П.1 — Архитектура проекта
**П.1.1** Использовать модульную структуру backend:
```
code/tongshu/
├── core/          # monthly_calendar, day_detail, four_pillars, tung_shu
├── qmdj/          # day_chart, hour_chart, interactive_chart
├── bazi/          # ten_gods, luck_pillars, day_master_strength
├── xkdg/          # daily_hexagram, hourly_structures, date_selection
├── feng_shui/     # flying_stars, directions, great_sun_formula
├── symbolic/      # constellations_28, belt_system, day_stars
└── zeri/          # search_engine, xkdg_rating
```

**П.1.2** PostgreSQL как основная БД, SQLite как fallback для offline.

**П.1.3** Предварительный расчёт всех данных в БД (1900-2100) для мгновенного отображения.

# РИСКИ (Р)

## Р.1 — Данные и алгоритмы
**Р.1.1** **Несоответствие QMDJ расчётов:** Наши расчёты Zhi Run дают 68.2% match с Excel. Классический метод принят, но пользователь может ожидать 100% совпадения с другими сайтами.
- *Решение:* Добавить Chai Bu как альтернативу + чётко документировать методологию.

**Р.1.2** **Отсутствие справочных данных:** Для 28 Constellations, Belt System, Day Stars (Good/Bad), Activity Recommendations нет готовых данных в БД.
- *Решение:* Наполнить справочники вручную (перевод с CN/EN на RU) или парсинг + перевод.

**Р.1.3** **Сложность XKDG Date Selection Rating:** Алгоритм рейтинга bazi-calculator.com неизвестен (закрытый исходный код).
- *Решение:* Разработать собственный алгоритм на основе видимых критериев или отложить эту функцию.

**Р.1.4** **Неполные Tong Shu описания:** Для русского языка нет полного набора описаний 12 Officers и 28 Constellations.
- *Решение:* Использовать EN как базу + перевод на RU. Приоритет: 12 Officers.

## Р.2 — Технические
**Р.2.1** **Производительность БД:** Добавление 10+ новых таблиц к существующим 355K строкам.
- *Решение:* Агрегатная таблица t_tung_shu_daily (1 строка = 1 день) для быстрого отображения. Детальные таблицы — по запросу.

**Р.2.2** **Сложность визуализации:** 9-палатные сетки, круговые диаграммы, цветовая кодировка требуют сложного фронтенда.
- *Решение:* MUI Grid для 9-палатных сеток. Recharts/D3 для круговых диаграмм. CSS-in-JS для цветов.

**Р.2.3** **Расчёт персонализации:** Для каждого профиля нужен отдельный скоринг.
- *Решение:* Скоринг считать на лету (быстро) или кэшировать в t_day_score.

**Р.2.4** **Moon phases / Solar term точные времена:** Требуют астрономических расчётов.
- *Решение:* Использовать библиотеку skyfield или ephem для Python. Предварительный расчёт в БД.

## Р.3 — Проектные
**Р.3.1** **Scope creep (расползание задачи):** Слишком много функций из 3 сайтов.
- *Решение:* Строгая приоритизация по фазам. Phase 1 = минимум для работы. Остальное — поэтапно.

**Р.3.2** **Интеграция с существующим кодом:** Риск сломать текущий Ba Zi / QMDJ калькулятор.
- *Решение:* Новый модуль `tongshu/` — изолирован от существующего кода. API — чёткие контракты.

**Р.3.3** **Мультиязычность:** Перевод 100+ показателей на 2-3 языка — большой объём.
- *Решение:* Начать с русского. EN — Phase 2. ZH — Phase 3.

**Р.3.4** **Время разработки:** 5 фаз × 3-4 недели = 15-20 недель.
- *Решение:* Agile-подход. Быстрые итерации. MVP за 4-6 недель.

## Р.4 — Пользовательские
**Р.4.1** **Разные методологии:** mingli (Zhi Run 4 layouts) vs CMS (Convention 2) vs BaziCalc (своя методология) — результаты различаются.
- *Решение:* Задокументировать используемую методологию. Предоставлять ссылки на источники.

**Р.4.2** **Субъективность скоринга:** Оценка благоприятности зависит от интерпретации.
- *Решение:* Прозрачный алгоритм. Показывать составляющие скоринга (10 Gods: +5, Na Yin: -3, etc.).

**Р.4.3** **Профиль пользователя:** Результаты сильно зависят от точности DOB.
- *Решение:* Валидация ввода. Предупреждение о неизвестном часе рождения.
